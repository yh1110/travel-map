# iPhone実機での動作確認 — 着手前チェックリスト

## Rule

iPhone実機(シミュレータではなくUSB/Wi-Fi接続した端末)でExpo dev-clientを動かす前に、必ず以下2点を確認・実施する。省略すると「スポットの取得に失敗」「接続が切れる」を無限に繰り返すことになる。

### 1. Metroは`localhost:8081`(ネイティブのデフォルトポート)で起動する。他ポート + 起動時引数トリックは使わない

**やってはいけないこと**: ポート8081が別プロジェクトと衝突するたびに、空いている別ポート(8082, 8085等)でMetroを起動し、`xcrun devicectl device process launch -- -RCT_jsLocation "<ip>:<port>"`という起動時引数でNSUserDefaultsを上書きして接続先を教える、という場当たり的な回避を繰り返さない。

**理由**: この起動時引数によるNSUserDefaults上書きは**そのプロセス起動1回限りの一時設定**。アプリがバックグラウンド/フォアグラウンド切り替えやOSによる再起動で再度起動されるたび(こちらが`devicectl`で明示的に起動していないケースも含む)に消え、React Native側は`RCTBundleURLProvider`のデフォルトフォールバックである`localhost:8081`を見に行く。もしそこに何もいなければ「No script URL provided」、別プロジェクトのMetroがいれば「Unable to resolve module」の赤画面になる。これが「毎回接続が切れる」の正体。

**正しい手順**:
1. `lsof -nP -i :8081 | grep LISTEN` で8081が空いているか確認する
2. 埋まっている場合、それが自分のプロジェクトのものでなければ、ユーザーの許可を得た上で該当プロセスをkillする(他プロジェクトのプロセスを勝手に殺さない)
3. 空いたら`npx expo start -p 8081`(または`expo run:ios -p 8081 --device`)で必ず8081を使う。これで起動時引数の指定が一切不要になり、アプリがどんな理由で再起動されても自動的に正しいMetroに繋がる

### 2. worktreeをまたいで`.env`をコピーする際、`EXPO_PUBLIC_SUPABASE_URL`のホスト部分を確認する

**やってはいけないこと**: 新しいworktreeに`.env`が無いとき、既存worktree(多くはシミュレータ用に`127.0.0.1`のまま運用されている)の`.env`をそのままコピーして終わりにしない。

**理由**: `127.0.0.1`はシミュレータからは常にMac自身を指すため機能するが、**実機からは実機自身のlocalhostを指してしまい、Macで動いているSupabase(Docker)に絶対に到達できない**。「スポットの取得に失敗しました: fetch failed」「サインインに失敗しました」は99%これが原因。

**正しい手順**: 実機向けに`.env`を用意する際は、`EXPO_PUBLIC_SUPABASE_URL`のホストを`ipconfig getifaddr en0`で取得したMacのLAN IPに書き換える。書き換えたら**Metroの再起動が必須**(dotenvはMetro起動時にのみ読み込まれ、既存プロセスには反映されない)。

## Why

2026-07-21、実機テストで上記2点を毎回見落とし、接続エラー・Supabase接続エラーを繰り返し発生させ、ユーザーを強く怒らせた。個人のメモリ(セッション横断の記憶)ではなく、このリポジトリの`.claude/rules/`に置くことで、どのworktree・どのcmuxペインでこのプロジェクトを開いても、Claudeが最初に読む場所に確実に情報が残るようにしている。

## How to apply

実機ビルド・起動の作業に入る前に、上記2点を着手前チェックリストとして先に処理する。後から気づいて都度その場しのぎで直す、を繰り返さない。
